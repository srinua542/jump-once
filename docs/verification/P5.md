## 2. Exit-gate evidence

- **Gates reject below-threshold fixtures and pass above-threshold fixtures:** each of the three gates has a passing fixture and a failing fixture, and each *individual* metric has a targeted failure case (`GdosEmotional`, `GdosStreamability`, `GdosInfoDensity`).
- **Every gate traces to a PRD score:** every `MetricScore.metric` is a §6/§7 metric name — `curiosity`/`confidence`/`surprise`/`mastery` (REQ-055), `shareability`/`clipPotential`/`reactionDensity`/`replayValue` (REQ-056), `peakScreenDensity`/`failureVisibility` (REQ-061/REQ-016) — and each `GdosReport` is stamped with the `profileId` it was scored under, because cross-version scores are not comparable (dm-0031).
- **Design Memory records each decision with full Intent Repository fields:** `parseDesignMemory` enforces all five §12 fields, non-empty, on every one of the ledger's 42 entries; the store round-trips the live file byte-for-byte. It is *dogfooded* — dm-0040, dm-0041 and dm-0042 were appended to the real ledger **through the store itself**, not by hand.
- **Calibration is external (the load-bearing property, dm-0031):** gate logic contains zero calibration literals. Three independent tests score one bundle under two profiles and assert the verdict flips while the score is identical. Recalibration is a data change; CDRE is its governed channel.
- **No re-auditing, and gdos is pure (dm-0031/dm-0037):** two scans in `EvalIsolation.test.ts` prove `src/eval/gdos/` imports the P4 local pass only as `import type`, and imports none of `AgentHarness`/`Evaluate`/`local/Search`/`core/Engine`. The two modules that *execute* a sim or search (`Evaluate.ts`, `EmergentFun.ts`) live at top-level `src/eval/`. The math whitelist is scan-enforced across `src/eval/gdos/`.
- **Determinism:** every scorer, estimator, miner and store operation is pure — no `Math.random`, no `Date.now` (the Design Memory takes its date as a parameter; CDRE takes its inputs as parameters), no delta-time. `GdosEvaluate` asserts an identical report across two full assembler runs; `EmergentFun` asserts an identical probe report; `GdosCdre` asserts identical mining.

## 4. Subtractive Removal pass (REQ-022 — run with the S5.7 engine itself)

The mandatory milestone subtraction pass was executed by `subtractivePass()` over all sixteen P5 modules, each held against the six pruning questions. **It found real debt.** Three exports had zero call sites anywhere in `src/` or `test/`:

| Removed | Where | Why it was debt |
|---|---|---|
| `CDRE_BASE_PROFILE` | `gdos/Cdre.ts` | An unused alias for `DEFAULT_PROFILE` — a second name for one thing (fails *non-duplicative*). |
| `TICK_SECONDS` | `eval/Evaluate.ts` | An unused re-export of `FIXED_STEP_SECONDS` — a second name for the one normative constant (fails *non-duplicative*). |
| `runFor` | `gdos/Evidence.ts` | An unused accessor written in anticipation of a caller that never arrived (fails *degrades-if-removed*). |

After removal, the pass re-ran clean: **16 items held, 16 kept, 0 removal candidates, 0 incomplete.** The suite stayed 336/336 green. Recording this is the point of the exercise — a subtraction pass that never removes anything is a rubber stamp, which is exactly the failure mode `subtractivePass` guards against by refusing to treat an unanswered question as a keep.
