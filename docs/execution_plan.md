## P1 ÔÇö Deterministic Core Architecture

### Work
Build the domain-agnostic deterministic runtime: a pure `Read Ôćĺ Process Ôćĺ Emit` state processor with a single source of truth, fixed-timestep simulation, and seeded determinism. This is the substrate every later system plugs into; it must be provably deterministic and immutable before any gameplay rides on it.

### Design summary (the compact model the code is written from)
1. **State is one immutable tree.** `GameState<TWorld>` holds `tick`, `clock`, `rng`, `input`, and a domain `world`. The core is generic over `TWorld` so it is testable with zero gameplay code present.
2. **Systems are pure step functions.** `System.step(state) => state`. No system holds state between frames or reads another system's internals; all shared state is in the tree.
3. **One mutation point.** `StateManager.commit()` is the only place the current snapshot is swapped; it retains the previous snapshot for render interpolation and can deep-freeze in tests to catch accidental mutation.
4. **Fixed-step determinism.** `Clock` banks variable real delta and emits a whole number of fixed steps; `Rng` is seeded and threaded through state. Same `(seed, input-tape)` Ôçĺ identical state, bit for bit.

### Dependencies
P0 governance exists; toolchain builds.

### Deliverables
- `src/core/Vec2.ts` (**done**), `src/core/Rng.ts` (**done**), `src/core/Clock.ts` (**done**), `src/core/State.ts` (**done**), `src/systems/System.ts` (**done**), `src/core/StateManager.ts` (**done**).
- `src/core/Engine.ts` ÔÇö the loop that pulls fixed-step count from `Clock`, threads systems, advances `tick`, commits via `StateManager`. (**done** ÔÇö S1.7)
- `test/unit/*` ÔÇö one suite per core module. (**done** ÔÇö S1.8, 55 unit tests)
- `test/integration/DeterministicReplay.test.ts` ÔÇö determinism proof. (**done** ÔÇö S1.9, 5 tests)
- `docs/verification/P1.md` ÔÇö verification report. (**done** ÔÇö S1.10)

### Validation criteria
- **Determinism:** two independent runs of the same seed + input tape produce an identical serialized final state. *(REQ-121)*
- **Immutability:** committing under `freezeOnCommit` throws on any in-place mutation attempt; no test mutates a prior snapshot. *(REQ-P10)*
- **Fixed step:** feeding `N ├Ś FIXED_STEP_SECONDS` of real time yields exactly `N` simulation steps; sub-step time banks correctly; a huge delta clamps (no spiral of death). *(REQ-121, REQ-160)*
- **Idempotency:** committing the same snapshot twice changes nothing but `previous`. *(REQ-P08)*
- **Isolation:** `src/core` imports nothing from `src/systems`, `src/entities`, or gameplay. *(REQ-P09)*
- **Green build:** `npm test` exits 0 with every core module covered.

# P2 ÔÇö Data Models & Level Definition Schema  *(CLOSED ÔÇö VERIFIED, `docs/verification/P2.md`; authored at S2.1 start per REQ-P02)*

## Work

Give the deterministic core something real to process: the pure-data component vocabulary (`src/components/`), the serialized Level Definition Schema (PRD ┬ž13 ÔÇö tilemap, entities, constraints, triggers, GDOS metadata), the loader/validator that is the only path from untrusted input to typed data, and the deterministic instantiation that turns a parsed level into the engine's `WorldState`. P2 ends when a hand-authored level file parses, validates, round-trips losslessly, and drives the engine deterministically.

This plan was produced by an adversarial architecture review of the P2 data model (challenges Ôćĺ resolutions). The decisions below are recorded in the design ledger as `dm-0009`ÔÇô`dm-0012`.

## Design summary (the compact model the code is written from)

1. **Static/dynamic split (dm-0009).** The tilemap holds only permanently-static geometry; the loaded `LevelDefinition` is deep-frozen once and reference-shared by every snapshot ÔÇö never copied per frame. Everything that can change at runtime (collapsing floors, doors, moving platforms) is an *entity* with runtime state, never a tile. "Modifying the layout" (┬ž16) is an entity state transition.
2. **Stable identity.** Entities carry branded `EntityId` strings; the validator enforces uniqueness and referential integrity (every trigger target resolves). Runtime spawns draw ids from a deterministic counter *inside* `WorldState` (`rt:<serial>` namespace, disjoint from authored ids) so spawning is a pure, replayable state transition.
3. **Entities are records, not classes and not an ECS.** `EntityDef = { id, transform, collider, behavior }` where `behavior` is a closed discriminated union matching the ┬ž16 library ÔÇö `behavior.kind` is the single discriminant (no redundant top-level `kind` field, so an entity/payload mismatch is unrepresentable instead of a validator rule) (moving platform, collapsing floor, ice, spike, laser, moving hazard, plate, proximity zone, door, spring, gravity zone, conveyor). Systems (P3) key on component presence, not entity kind. Sparse-set/archetype ECS machinery is rejected as over-engineering at this entity count; entity classes are rejected as a data/logic-decoupling violation.
4. **Serialization is canonical JSON (dm-0010, hardened by dm-0014).** Finite IEEE doubles round-trip exactly; therefore the validator rejects non-finite numbers and normalizes `-0` to `0`, and `serializeLevel()` emits a fixed field order. Unknown/extra keys are **strictly rejected** at every level of the tree ÔÇö a lenient parser would re-serialize a future-version file with its fields silently dropped, a "lossless" round-trip that loses data (dm-0014). Round-trip safety is defined precisely: value-level `deepEqual(parse(serialize(v)), v)` plus byte-level idempotence `serialize(parse(serialize(v))) === serialize(v)`. Byte-identity with the hand-authored source file is deliberately not required. The canonical form itself is anchored by a committed golden hash of the serialized sample fixture, so an accidental change to key order or number formatting fails a test instead of invalidating every stored hash downstream.
5. **Versioning is a policy, not a framework (dm-0010).** `schemaVersion: 1` is required at the root; the loader hard-rejects any other value ÔÇö no best-effort parsing. Migrations are pure `vN Ôćĺ vN+1` functions written only when v2 exists. Zero migration code ships in P2, by decision.
6. **Parse, don't validate ÔÇö and it lives in `src/schema/` (dm-0013).** `parseLevel(raw: unknown): Result<LevelDefinition, SchemaError[]>` is the only construction path from untrusted input; errors carry JSON-pointer-style paths. Placement: the serializer/validator/loader are *function bodies* and therefore cannot live in logic-free `src/components/` (the S2.1 scan test would fail them), are not per-frame systems (`src/systems/`), and are Jump-Once-specific (not `src/core/`). They get a dedicated `src/schema/` directory (directory contract updated); the schema *types* stay in `src/components/`; `instantiateWorld()` stays in `src/entities/`. The validator is hand-rolled and dependency-free; the drift risk is priced by one failing-fixture test per rejection rule **plus a seeded fuzz test** (mutations of the valid fixture via `src/core/Rng` ÔÇö wrong types, nulls, truncations, injected keys ÔÇö must always yield a `Result` error, never a throw or a false accept).
7. **GDOS metadata: structural now, semantic in P5 (dm-0012).** The ┬ž13 block is typed structurally minimal but complete ÔÇö `targetKgNode`, `difficultyVectors` (named-axis record), `emotionalBudgetCurve` (keyframes of the four ┬ž6 metrics over normalized level progress), `creatorMomentFrame` (`{tickWindow, description}`). P2 validates structure (finiteness, [0,100] ranges, strictly increasing keyframes); P5 owns semantics (thresholds, scoring). If P5 needs more, that is a schema-version bump plus one migration function.
8. **Data-driven values Ôëá data-driven axioms (dm-0011).** The constraints block carries bounds, spawn, goal, and par-time tiers. The single-jump rule is an engine invariant (REQ-004, P3), never a schema parameter ÔÇö a `maxJumps` field would put the game's identity one typo from falsification.
9. **Coordinate convention** is decided once at S2.2 and ledgered. Recommendation going in: y-down, origin top-left, gravity +y, `tileSize` in world units (matches row-major tilemap authoring and screen space; only P3 physics intuition pays a small, localized tax).

## Adversarial review ÔÇö findings and upgrades over the original slice table

1. **Missing slice: run lifecycle (goal / defeat / scene reload).** No original slice owned goal-reach detection, the defeat state, or scene reload ÔÇö yet REQ-010/011's acceptance ("jump refreshes *only* on scene reload") is untestable without a reload mechanism, and REQ-003's "instant gameplay iteration" is precisely the defeatÔćĺreload loop. **Fix:** new slice S3.4 (run lifecycle) before the jump-lock slice.
2. **Input-system ownership was undefined.** `LevelInstantiation.test.ts` injects `InputFrame`s manually and noted "the input system doesn't exist until P3". **Decision (dm-0019):** the *simulation-side* input contract already exists ÔÇö `GameState.input: InputFrame` (S1.4); P3's player controller consumes it and nothing else. Live device capture (keyboard/touch Ôćĺ `InputFrame`) is render/SDK territory: P9 (REQ-170/171). Serialized replay tapes are P4's (assigned at P4 planning per the P2 open question). No new P3 slice needed; the boundary is now explicit.
3. **Integration technique upgraded: closed-form tick-parametric kinematics (dm-0016).** Movers (platforms/hazards), timed lasers, and collapse timers are *derived from the tick* (`position = pathPosition(elapsedTicks)`), never incrementally integrated. This eliminates float accumulation drift, makes every entity's state exactly reproducible at any tick, and makes scene reload trivially correct. Only the player (input-coupled) is integrated.
4. **Collision technique upgraded: swept, axis-separated resolution (dm-0017).** The original criterion said "AABB collision"; naive discrete overlap tunnels through thin tiles at spring-launch velocities. Player-vs-tilemap movement is resolved per axis by scanning the swept tile range and clamping to the first solid boundary ÔÇö correct at any speed, no epsilon nudges (contact snaps exactly to tile edges, which is float-exact). Player-vs-entity lethal/contact checks use the Minkowski-sum swept test where miss = exploit (hazards).
5. **Cross-engine float determinism hardened (dm-0017).** Simulation math is restricted to IEEE-754-exact operations: `+ Ôłĺ ├Ś ├Ě`, `Math.sqrt`, comparisons, `Math.min/max/abs/floor/ceil/trunc`. **Transcendental functions (`Math.sin/cos/tan/pow/exp/hypot/ÔÇŽ`) are banned in `src/systems/`** ÔÇö they are not correctly-rounded and vary across JS engines, which would silently break replay hashes between Node (P4 solvers) and browsers. Waypoint traversal uses arc-length arithmetic, not trig. Semi-implicit (symplectic) Euler is the integrator: velocity first, then position ÔÇö stable under constant gravity at fixed step.
6. **Jump-lock refresh is correct by construction (dm-0018).** All jump-lock state lives in `WorldState`; scene reload = `instantiateWorld(level)` (pure, already proven deterministic). The "refreshes only on reload" invariant therefore cannot be violated by a forgotten reset ÔÇö there is no reset code to forget.
7. **Tuning values are data, not literals (dm-0018).** Gravity, run speed, jump velocity, anticipation ticks etc. live in a frozen pure-data record `src/components/Tuning.ts` (logic-free; the S2.1 scan test covers it automatically). They are campaign-global by design ÔÇö per-level physics would fragment game feel; if GDOS ever demands per-level overrides, that is a schema-version bump (dm-0010 path). No `maxJumps`-like field exists anywhere (dm-0011 holds).
8. **Dependencies tightened.** Environmental elements (riding a moving platform, collapsing on stand) and triggers (plates need the player standing on them) require the *controller and grounding*, not just the physics core ÔÇö original table under-declared this. New table declares the true edges.
9. **Canonical system pipeline order is fixed data.** With seven systems, composition order is a determinism parameter. The order below is normative; the Engine pipes systems in exactly this order.
10. **Quadtree kept, made deterministic.** REQ-162 names a quadtree; intent is neighborhood-only collision queries. The tilemap needs no partition (the grid *is* one ÔÇö O(1) lookup); the quadtree covers *entities*: rebuilt per query set as a pure function, fixed insertion order (array order), fixed node capacity and max depth, so structure is replay-identical. Equivalence-vs-brute-force under seeded fuzz is the acceptance test.
11. **REQ-150 visual sub-clauses re-scoped.** Squash-and-stretch, particle burst, camera tracking are render-side; P3 owns the *simulation* share (anticipation ticks, lock machine, instant accel/decel) and exposes the state P9 animates from. ## Adversarial review ÔÇö findings and upgrades over the original slice table

1. **Replay-tape serialization was unowned (P2 open question #5, carried through P3) ÔÇö owned here, at S4.1 (dm-0023).** The harness is the first producer of tapes, so it defines the format: a versioned record `{ schemaVersion, levelId, seed, frames[] }` where each frame is one `InputFrame` per simulation tick. The full P2 serialization discipline applies verbatim (dm-0010/dm-0014): canonical field order, strict unknown-key rejection at every path, non-finite rejection, `Result`-typed parse with path-qualified errors, value-lossless + byte-idempotent round-trip. I/O lives in `src/schema/TapeIO.ts` (definition-time I/O ÔÇö the dm-0013 placement rule). P8's input recorder and P9's save layer consume this format later; they do not redefine it.
2. **Evaluation logic had no legal home in the directory contract.** Agents/audits are not per-frame systems (`src/systems/` runs inside the engine loop; agents drive it from outside), not schema I/O, not core, not components. **Decision (dm-0022): new `src/eval/` directory** ÔÇö evaluation-time logic that consumes the sim strictly through its public contracts (`createInitialState`, `Engine`, `StateManager`, `FIXED_STEP_SECONDS`, read-only `GameState`). The dependency is one-way: nothing under `src/core|systems|components|entities|schema` may import from `src/eval/` ÔÇö the sim must never know it is being judged. Directory contract updated alongside this plan.
3. **Live-run vs. tape-replay divergence risk: agent randomness must not touch the sim's RNG.** If a policy drew from `state.rng`, replaying the recorded tape (no agent present) would leave the sim RNG un-advanced and the final state would differ bit-wise ÔÇö silently breaking the P4 determinism contract. **Decision (dm-0024): agents thread their own seeded `RngState` stream** (derived from the run seed, threaded through agent memory); `state.rng` belongs to the simulation alone. Live-final-state Ôëí replay-final-state (bit equality) is an acceptance test *per archetype*.
4. **Agent statefulness must be explicit or determinism is fiction.** Policies are pure: `decide(state, memory) Ôćĺ { input, memory }`. All agent memory (held intent, hesitation countdowns, per-life plans, the agent RNG cursor) is threaded through the harness exactly like sim state ÔÇö no closures over mutable variables, no hidden fields. Same (level, seed, archetype) Ôçĺ identical tape, twice, by construction.
5. **Halting is a property, not a hope.** Every harness run carries a data-driven budget (`maxTicks`, `maxAttempts` ÔÇö an `EvalBudget` record, not literals). Budget exhaustion is a first-class `'timeout'` outcome, never an exception and never a hang. S4.3's softlock detector *builds on* classified timeouts; nothing in P4 can stall the future generation pipeline.
6. **Archetypes are data-parameterized policies, not five code forks (dm-0024).** One shared sensing/decision core; five frozen `ArchetypeParams` records (reaction cadence, hesitation ticks, jump-commit lookahead, hazard caution, exploration prefix). Behavioral distinctness is *proven by test* ÔÇö pairwise-distinct tapes on a discriminating fixture ÔÇö not asserted by naming. Adding a sixth archetype later is a data change.
7. **Archetype play is evidence, not proof ÔÇö solvability needs search (scoped to S4.2).** A reactive archetype failing a level does not prove the level unsolvable; REQ-141's solvability audit therefore gets a bounded deterministic search over input space at S4.2, with archetype runs as its fast path and witness generator. S4.1 deliberately ships the harness + archetypes only; conflating the two was the original table's hidden scope error.
8. **The two validation passes stay physically isolated (REQ-140).** Local Spatial Verification (S4.2ÔÇôS4.5, per-level) and Macro Curriculum Validation (S4.6, per-chapter) are separate modules with separate verdict types; the macro pass consumes local verdicts *as data* and never re-runs simulations internally. Isolation is structural (imports), not stylistic.
9. **Five-tier routing is grounded in archetype times (REQ-101/102).** Discovery Ôëł First-Time completion; World Record Ôëł Expert-Speedrunner best; the middle tiers interpolate from the archetype spread. Optimization delta = T_Discovery Ôłĺ T_WR; zero/minimal delta Ôçĺ reject. The authored `parTimeTiersSeconds` (S2.2) becomes a cross-check input, not the source of truth.
10. **The math whitelist extends to `src/eval/` (dm-0017 discipline).** Agent decisions produce tapes; tapes must replay bit-identically on any JS engine. Same audit as `src/systems/`: no transcendental functions, only `+ Ôłĺ ├Ś ├Ě %`, `Math.sqrt/min/max/abs/floor/ceil/trunc`.
11. **The axiom is P4's ground truth, not its parameter.** The solvability audit treats "exactly one jump" as engine-enforced fact (dm-0020) ÔÇö it never re-implements the lock, never counts jumps by parsing input (it reads `world.jumpLock`), and no `maxJumps`-like knob exists anywhere in eval config (dm-0011 holds).

## Adversarial review ÔÇö findings and upgrades over the naive slice table

1. **The core defect of a naive Campaign Director: mutable rolling state.** A director that updates fields in place violates dm-0004 (never mutate `GameState` in place ÔÇö this invariant generalises to all state records). A na├»ve implementation would keep a mutable `CampaignState` object and patch it field by field per level. **Decision (dm-0043): the campaign update is a pure fold step `updateState(state, record, profile) Ôćĺ CampaignState`, returning a new record every time.** The zero-state is `ZERO_CAMPAIGN_STATE` (the additive identity, same pattern as `Vec2.ZERO` in the engine). Processing a full campaign is `records.reduce(updateState_curried, ZERO_CAMPAIGN_STATE)` ÔÇö no mutation, no side effects, replay-identical.

2. **Behavioral signals must be frame-count based, not wall-clock based.** Replay tapes (`AgentRunResult`) carry frame-indexed input events, not real-time timestamps. Wall-clock derivation would (a) violate dm-0003 (fixed step, no delta-time scaling) and (b) break determinism across machines. **Decision (dm-0043): every behavioral signal in `BehaviorSignals` is expressed in frame counts.** Hesitation = consecutive no-input frames before a jump input; panic burst = N jump-or-move inputs in M consecutive frames; commitment speed = frame index of the first jump input; drop-off = tape ends without the goal-reached event. These are reproducible purely from the tape.

3. **P8 contract boundary: interface now, implementation later.** REQ-032 is co-owned by P6 and P8. The risk is coupling the `CampaignDirector` to a concrete telemetry collector that does not yet exist, so either (a) P6 ships half-baked pending P8, or (b) P8 forces a P6 refactor when it arrives. **Decision (dm-0044): P6 defines `BehaviorSignals` as a typed data record and `TapeAnalyzer` as the derivation path from replay tapes (the P8-independent route). The `CampaignDirector` depends only on `BehaviorSignals`, never on the source. P8 will supply live-collected signals as an alternative implementation of the same record shape ÔÇö a data substitution, not a structural change.** This is the "abstract interface" pattern the P5 verification report recommended (┬ž7 item 2).

4. **Coverage matrix consumed by reference, never duplicated.** P5 built `coverageMatrix(evidence[]) Ôćĺ CoverageMatrix` and owns its derivation. P6 tracks *campaign-level* introduction and mastery against it ÔÇö which axis cells have been activated across levels so far ÔÇö but does not re-derive or store the matrix definition. **Decision (dm-0045): `MechanicTracker` receives a `CoverageMatrix` as a parameter (passed in by the caller); it never imports or re-computes the matrix itself.** "Mechanics introduced" = at least one level record's evidence covers a cell for that mechanic; "mechanics mastered" = that mechanic appears in the top optimization tier AND the exploration-archetype trace shows unambiguous routing confidence (Ôëą profile threshold). One source of truth for the matrix; P6 is a consumer.

5. **`CampaignProfile` is a separate schema from `ScoringProfile`.** A tempting simplification is to extend `ScoringProfile` with campaign-level calibration ÔÇö spike thresholds, mastery weights, retention multipliers. This would couple two distinct concerns and force a `PROFILE_SCHEMA_VERSION` bump on `ScoringProfile` (dm-0039). **Decision (dm-0045): `CampaignProfile` is its own versioned data structure (`campaignProfileSchemaVersion: 1`), strict-parsed with the full P2 discipline, with its own `DEFAULT_CAMPAIGN_PROFILE` frozen constant.** The zero-numeric-literal validation criterion applies to P6 gate logic exactly as it does in P5: every threshold/weight reaches code through a parsed `CampaignProfile`; a two-profile test proves calibration is external.

6. **REQ-142 elevation: from per-session verdict to running campaign trajectory.** P4 computed the four macro criteria (Cognitive Structural Mapping, Cross-Chapter Degradation Analysis, Curiosity Progression Curves, Graduation Assessment Verification) for a single fixture curriculum run ÔÇö a one-shot `MacroCurriculumVerdict`. P6 must *track them as running state across the full campaign arc*, enabling cross-chapter degradation *trends* over time. **Decision (dm-0046): `ChapterHealth` ingests the chapter's `MacroCurriculumVerdict` records (P4 output, passed as data) and lifts each criterion to a trend line in `CampaignState`.** The P4 verdict is an input; P6 computes the *delta* between adjacent chapters and stores the trajectory as campaign state. This is the share of REQ-142 that P4 explicitly left open ("P6 contributes the campaign models").

7. **Retention prediction is a design proxy, not a psychological model.** Calling a field "retention prediction" risks scope inflation toward ML or player psychology. **Decision (dm-0046): `retentionPrediction` is a score in [0, 1] computed as a `CampaignProfile`-weighted composite of observable behavioral signals** ÔÇö high retry cadence with falling performance trends downward; high optimization depth with increasing commitment speed trends upward; panic cycles trend downward; curiosity trend and chapter health each contribute a weighted component. It is explicitly documented as a *proxy for whether the design is rewarding the player*, not a claim about player intent or future actions. The weights live in `CampaignProfile`.

8. **`CampaignDirector` is not a `gdos/` module.** The `gdos/` sub-tree is PURE over pre-assembled evidence ÔÇö it runs no sim, no search, no audit, and calls no external I/O (dm-0037; enforced by two scans in `EvalIsolation.test.ts`). The `CampaignDirector` operates at a *higher level of abstraction* than per-level scoring: it consumes `GdosReport`s (already judged by P5) and drives the fold over a campaign record sequence. **Decision (dm-0047): all P6 code lives in `src/eval/campaign/` ÔÇö a new top-level sub-folder of `src/eval/`, alongside `Evaluate.ts` and `EmergentFun.ts`, not inside `gdos/`.** The `EvalIsolation` scan's rule ÔÇö `gdos/` must not import the audit entry points or the engine ÔÇö is unaffected. `campaign/` is a consumer of `gdos/` output data types, not of its internal computation.

9. **Spike detection must use profiled thresholds, not heuristics.** A difficulty spike is "chapter health drops sharply from the rolling baseline." "Sharply" must not be a hardcoded constant. **Decision (dm-0048): the spike detector in `CampaignDirector` computes a rolling `chapterHealthBaseline` (exponential moving average, decay factor in `CampaignProfile`) and flags a spike when the current chapter's health falls below `baseline Ôłĺ profile.spikeDropThreshold`.** Flagged spikes are emitted as typed `CampaignAlert` records ÔÇö same pattern as `DesignDecision` in P5 ÔÇö with a reason string and the contributing metrics. No bare booleans; the alert carries evidence.

10. **Alternatives weighed and rejected:** a mutable event-stream model (observer/subscriber pattern) would allow decoupled updates but introduces non-determinism in subscriber order and violates the stateless-system invariant; a single monolithic `CampaignAnalyzer` module (violates per-concern testability and makes it impossible to unit-test behavioral extraction independently from chapter health); a JSON-rules engine for spike/retention logic (interpreter accidental complexity ÔÇö the thresholds are data in `CampaignProfile` already, the logic is simple enough to be TS expressions); importing P4 audit functions directly from `src/eval/local/` in P6 (would re-run the sim ÔÇö P6 consumes P4 output records passed as data, never re-runs audits).

## Design summary (the compact model the code is written from)

1. **Kernel (`src/eval/campaign/CampaignState.ts`).** `CampaignState` ÔÇö the ten macro variables as typed fields: `knowledgeState` (per-mechanic confidence), `behaviorState` (`BehaviorSignals` rolling aggregate), `emotionalState` (inferred from panic/commitment/retry), `skillCurve` (rolling performance delta), `mechanicsIntroduced` (set of mechanic keys), `mechanicsMastered` (set of mechanic keys), `optimizationDepth` (mean tier reached across levels), `curiosityTrend` (exploration divergence direction), `chapterHealth` (map of chapterId Ôćĺ `ChapterHealthReport`), `retentionPrediction` ([0, 1] composite). `ZERO_CAMPAIGN_STATE` is the additive identity. `CampaignProfile` ÔÇö calibration record: mastery threshold, spike drop threshold, baseline decay factor, retention component weights, behavioral signal sensitivity thresholds ÔÇö strict-parsed, `DEFAULT_CAMPAIGN_PROFILE` frozen.

2. **Behavioral extraction (`src/eval/campaign/TapeAnalyzer.ts`).** `analyzeTape(tape: AgentRunResult, profile: CampaignProfile) Ôćĺ BehaviorSignals`. Signals: `hesitationFrames` (max gap before a jump input), `retryCount` (tape sequence length Ôłĺ 1), `panicBurstCount` (clusters of N inputs in M frames), `commitmentSpeed` (frame index of first jump), `platformCheckCount` (lateral moves near-edge without jump), `dropOff` (goal event absent). Every signal derived from frame-indexed events only ÔÇö no wall-clock, no `Math.random` (dm-0043).

3. **Mechanic tracking (`src/eval/campaign/MechanicTracker.ts`, `KnowledgeModel.ts`).** `trackMechanics(state, record, coverageMatrix, profile) Ôćĺ { mechanicsIntroduced, mechanicsMastered }`. Introduced: the level's `GdosReport` covers Ôëą 1 coverage matrix cell for that mechanic. Mastered: top optimization tier reached AND exploration-archetype routing confidence Ôëą profile threshold (dm-0045). `KnowledgeModel.updateKnowledge(knowledgeState, record, profile) Ôćĺ knowledgeState` ÔÇö per-mechanic confidence updated via the evidence in the `GdosReport`.

4. **Chapter health (`src/eval/campaign/ChapterHealth.ts`).** `aggregateChapterHealth(records: LevelRecord[], macroCriteriaVerdicts: MacroCurriculumVerdict[], profile: CampaignProfile) Ôćĺ ChapterHealthReport`. Incorporates all four REQ-142 criteria elevated to campaign trajectory (dm-0046): Cognitive Structural Mapping (each mechanic's prior-mechanic dependency chain respected), Cross-Chapter Degradation (performance delta vs previous chapter), Curiosity Progression Curves (curiosityTrend direction across chapter), Graduation Assessment Verification (chapter-final level demonstrates mastered mechanics). Returns `{ score, criteriaScores, trend, alerts }`.

5. **Campaign Director (`src/eval/campaign/CampaignDirector.ts`).** `updateState(state: CampaignState, record: LevelRecord, chapterMap: ChapterMap, profile: CampaignProfile) Ôćĺ CampaignState` ÔÇö the core fold step (dm-0043). `processCampaign(records: LevelRecord[], profile: CampaignProfile) Ôćĺ CampaignReport`. Spike detection: rolling `chapterHealthBaseline` (EMA, decay factor from profile) ÔÇö emits a `CampaignAlert` when health drops below `baseline Ôłĺ profile.spikeDropThreshold` (dm-0048). Retention prediction: `CampaignProfile`-weighted composite (dm-0046). Emotional state: inferred from `panicBurstCount`, `commitmentSpeed`, `retryCount` per profile weights. Skill curve: frame-count performance delta over a rolling window. Curiosity trend: exploration-archetype divergence direction over last N levels (N from profile).

6. **`LevelRecord`** ÔÇö the P6 input unit: `{ levelId: string; chapterId: string; report: GdosReport; tape: AgentRunResult; macroCriteria: MacroCurriculumVerdict }`. Assembled by callers from P4 + P5 outputs. P6 code never imports `AgentHarness`, never drives the engine, never re-runs P4 audits. `CampaignReport` ÔÇö output: `{ finalState: CampaignState; chapterHealthMap: Map<string, ChapterHealthReport>; alerts: CampaignAlert[]; retentionPrediction: number }`.

10. **Alternatives weighed and rejected (dm-0054ÔÇôdm-0060):** grammar/wave-function-collapse/ML generators (unverifiable or nondeterministic; parametric archetype templates are auditable and sufficient for pipeline verification ÔÇö P10 can extend the template library without structural change); modifying `Search.ts` to strip jump actions from `ACTIONS` (the `forbidden` predicate achieves the same with zero touch of verified P4 code); a standalone lifecycle store (two sources of truth ÔÇö the ledger already reserved the registry); placing the pipeline under `tools/` (loses purity + unit-test coverage; `tools/` is interactive P8 territory); extending `ScoringProfile`/`CampaignProfile` with gen calibration (couples three concerns; forces version bumps on verified schemas).

2. **Profiling's wall-clock tension with the determinism axiom.** REQ-132 wants frame-rate/timing measurement, but CLAUDE.md's determinism invariant forbids delta-time scaling and the simulation core never reads a wall clock (dm-0003/dm-0004 scope `FIXED_STEP_SECONDS` and immutable-commit discipline to `src/core`/`src/systems`/gameplay state). ## Design summary (the compact model the code is written from)

1. **Editor draft state (`tools/level_editor/EditorState.ts`).** An `EditorDraft` wraps a mutable-by-replacement `LevelDefinition`-shaped working copy plus an `UndoStack` of inverse-command pairs. Commands ÔÇö `paintTile`, `placeEntity` (grid-snapped), `groupEntities`, `ungroup` ÔÇö are pure functions `(draft, command) => draftÔÇ▓` that also push their inverse onto the stack; `undo`/`redo` pop/replay inverses, never re-derive state by other means. `exportDraft(draft) Ôćĺ Result<LevelDefinition, SchemaError[]>` is a thin call into the existing P2 `parseLevel`/`serializeLevel` pair ÔÇö the draft is only ever "real" once it round-trips the strict parser, so an invalid draft cannot export.

## Adversarial review ÔÇö findings and upgrades over the naive slice table

1. **The renderÔćötools direction (the handoff's explicit open question, deferred by the PKG's pending nodes).** The naive table never says where rendering lives, and the PKG's three `render/*` PlannedSystemNodes provisionally say `src/render/` while explicitly deferring the decision here. Every existing home fails: `src/` cannot import `tools/` (dm-0066 one-way rule), so an editor UI under `src/render/` could not read the `EditorDraft`/descriptor types it must render; `tools/` is scan-forbidden from canvas/DOM/WebGL (dm-0065, `ToolsIsolation.test.ts`) and retrofitting rendering there would break the split that let P8 close honestly. **Fix (dm-0081): a new top-level `render/` subtree, one-way-isolated at the TOP of the dependency chain.** `render/` may import `src/` (through the same public seams every other consumer uses) and `tools/` (as a typed library ÔÇö descriptors, editor state, inspector, profiler); **nothing under `src/` or `tools/` may ever import `render/`**; within the layer, `render/tooling/` is the *only* area permitted to import `tools/`, so the game shell stays tools-free. A `RenderIsolation.test.ts` scan lands at S9.1, the first slice to add a `render/**/*.ts` module ÔÇö the dm-0053 lesson (never let an isolation rule go prose-only) applied proactively, fourth subtree in a row. `directory_structure.md` and `tsconfig.json`'s `include` gain the subtree at S9.1; the PKG pending nodes materialize under the corrected paths.
2. **The canvas-vs-WebGL surface (deferred by the bible ┬ž6).** REQ-170 and the IRD name a **WebGL** renderer with batching; the bible's performance contract (┬ž5) is written in Canvas2D idiom (cached canvases, `drawImage` blits, one background bitmap, source-rect crop) because the Paper Collage engine *generates* art by 2D path drawing. These are not in conflict ÔÇö they describe two different moments. **Fix (dm-0082): Canvas2D is the load-time rasterizer; WebGL2 is the frame loop.** The StylePack rasterizes procedural art into cached bitmaps at exactly the bible's three sanctioned moments (asset creation, level load, appearance change) via an injected 2D raster device; cached bitmaps are packed into texture-atlas pages and the composed per-level background becomes one texture; the frame loop is batched textured quads ÔÇö the bible's "`drawImage` of cached canvases" maps 1:1 to "textured-quad draws of cached rasters," "one background bitmap + source-rect crop" maps to one background texture drawn with UV crop, and "generate never per-frame" survives verbatim. WebGL2 is the 2026 evergreen baseline; weighed and rejected: a Canvas2D frame loop (satisfies the bible but not REQ-170's wording, and forfeits instanced batching REQ-162 names), a WebGL1 fallback (permanently-dead code in 2026), a WebGPU backend (REQ-170 names WebGL; portal long-tail hardware; the device seam keeps a future backend additive, not a rewrite).
3. **The naive table silently drops P9's inherited shares and the art canon's own deliverable.** REQ-130/131 (render the editor + overlays, dm-0065) and REQ-132 (asset-delivery-speed profiling, dm-0072) appear in **no** naive slice, and neither do REQ-150's visual sub-clauses (P3,P9 column since dm-0008), REQ-016's visual share, REQ-001/002 ÔÇö nor the **PaperStylePack itself** (the naive table predates the canon; "visual grammar style system" as one slice conflates the style-agnostic grammar schema with StylePack #1's thousands of lines of procedural drawing). **Fix: explicit slices ÔÇö S9.2 (PaperStylePack port), S9.7 (asset pipeline + delivery profiling), S9.9 (editor/debug UI), S9.5 (game feel + fairness visuals) ÔÇö and REQ-001/002/171 folded into the shell slice S9.8 where they factually land.**
4. **Data-driven order inversion.** The naive table builds the WebGL renderer (S9.1) *before* the visual grammar (S9.3), guaranteeing the renderer invents ad hoc visuals the grammar must retrofit ÔÇö the "gameplay values hardcoded in logic" anti-pattern in visual form. It also makes WebAudio depend on the WebGL renderer, which it does not need. **Fix: grammar schema + StylePack interface are S9.1 ÔÇö data the renderer consumes from its first line** (dm-0083). Audio (S9.6) depends on the grammar + observable state transitions, not on WebGL. **The grammar/pack split (dm-0083):** the *grammar* ÔÇö the six REQ-071 categories, the `behavior.kind Ôćĺ category` bindings, criticality (Danger is always critical), and the signature-role contract (palette ┬Ě silhouette ┬Ě motion ┬Ě audio) ÔÇö is **style-agnostic, versioned, strict-parsed data** (the sixth versioned schema, after `ScoringProfile`/`CampaignProfile`/`GenProfile`/`TelemetryProfile` and the level schema itself); the *materials* (paper palette hexes, slab generators, tile kits, synth patches) live behind the StylePack. REQ-070's mixing prohibition is enforced **twice, structurally**: the grammar parser rejects any binding that maps one kind to two categories, and pack validation rejects any pack whose per-category signatures overlap or are incomplete. A future StylePack replaces materials only; grammar, bindings, level data, and tapes are untouched (dm-0076's mandate, made testable).
5. **Grammar reconciliation 1 ÔÇö Optimization has no dedicated accent (bible ┬ž1, flagged for this plan).** Adding a seventh accent colour strains a deliberately tiny palette and violates the kit's restraint; keeping terracotta glyphs (the kit's current conveyor chevrons) collides with Danger ÔÇö a terracotta accent on a non-lethal conveyor is actively misleading under REQ-016. **Resolution (dm-0084): Optimization's signature is formalized as an achromatic motion-glyph layer.** Its four channels: palette = **ink/cream glyphs only** (no chromatic accent ÔÇö the *absence* of colour plus motion IS the palette signature; no other category is achromatic-kinetic), silhouette = chevron/`Ôćö` arrow glyph family, motion = directional sweep, audio = directional whoosh patch. The kit's terracotta chevrons are recoloured ink/cream in the PaperStylePack port; terracotta glyphs become a pack-validation error. Colour-blind-safe by construction (motion + glyph shape carry the meaning; REQ-016's "never colour alone" rule). Rejected: a dedicated new accent (palette inflation for the least-common category; every candidate hue collided with an existing meaning or read as decor).
6. **Grammar reconciliation 2 ÔÇö Danger doubles as the goal (bible ┬ž1, flagged for this plan).** Terracotta currently marks both "kills you" and "the flag," separated only by silhouette. REQ-070 is strict and unyielding: one signature, one meaning; and REQ-016's fairness demands terracotta mean *exactly* "lethal." **Resolution (dm-0085): the goal unbinds from Danger and binds to Interactive (dusty teal).** Taxonomically honest ÔÇö sim-side, the goal *is* the terminal proximity trigger, and Interactive already means "logic that responds to your arrival" (zones, triggers, watchers, plates); emotionally, teal reads calm/inviting against terracotta threat; colour-blind players get silhouette (pennant vs. teeth) *and* colour separation. Danger's meaning tightens to exactly "kills you." Rejected: keeping the terracotta goal on silhouette-only separation (the bible's own "commitment target" unification is genuinely poetic for a one-jump game ÔÇö the jump commits you toward death or the goal alike ÔÇö but it is a signature mix under REQ-070's strict wording and dilutes REQ-016's "terracotta = failure information"); binding the goal to Secret/lavender ("exits" appear in Secret's meaning, but the *main* goal is neither hidden nor a rule-change, and a dashed "idea of a thing" outline on the primary objective misleads). Both reconciliations are **data-level** decisions (grammar bindings + pack materials) ÔÇö cheap to revisit if the art direction owner overrules; they are recorded in the bible at S9.1 (its two "open" flags close, exact-string edits).
7. **Real time vs. the determinism axiom.** A render loop needs `requestAnimationFrame` and wall-clock deltas, but dm-0067 scopes wall-clock reads to `tools/profiler/` only, and the axiom forbids real time influencing a tick. **Fix (dm-0082): the exception extends to the render layer with a precise contract ÔÇö real time enters simulation *only* as the `realDeltaSeconds` argument of the P1-designed `Clock.advance` seam (which banks it into whole fixed steps), and is otherwise confined to `render/platform/` + frame-time diagnostics. Rendering is a pure projection `(state, previousState, interpolationAlpha, grammar, pack, viewport) Ôćĺ DrawList`; nothing computed by the renderer is ever written into `WorldState` or any replay-checked value** (`interpolationAlpha` read-only ÔÇö dm-0004, extended per the bible ┬ž0). The wall-clock scan amends from "only `tools/profiler/`" to "only `tools/profiler/` + `render/`", with `src/` remaining scan-clean. The *visual* RNG is a seeded generator hashed off entity identity/grid position (bible ┬ž4) ÔÇö same discipline as the sim RNG; `Math.random` stays forbidden repo-wide.
8. **Headless verifiability of a browser-facing layer.** The DoD requires `node --test` proof and no browser exists in the test loop. **Fix (dm-0086): hexagonal device seams.** Every render-layer module is pure logic against injected interfaces ÔÇö `Raster2D` (the minimal 2D-context subset the PaperStylePack draws with), `Gl2Device` (minimal enumerated `WebGL2RenderingContext` subset), `AudioDevice` (subset of `AudioContext`), `AssetFetcher` (fetch-like), `PortalSdk`, `FrameScheduler` (rAF-like) ÔÇö with hand-rolled recording fakes in tests (established project culture: no jsdom, no headless-gl, no mocking framework). The only code touching real browser globals is thin binding modules under `render/platform/` (canvas context creation, rAF, script-tag SDK global, fullscreen), which contain no logic beyond construction and are the *only* files allowed to name browser globals ÔÇö scan-enforced. Pack tests assert *structure*, not pixels: identical command streams for identical identity seeds, byte-identical base geometry across states (dm-0078), no wear/grain commands on interiors or internal edges (dm-0080), cache keys hit on repeat requests (generate-never-per-frame). Real-browser smoke rides P11's build pipeline (open question 3).
9. **Poki SDK ÔÇö current official approach, verified 2026-07-11 against sdk.poki.com.** The SDK loads exclusively from Poki's CDN (`https://game-cdn.poki.com/scripts/v2/poki-sdk.js` ÔÇö server-pushed updates; bundling is not the supported path), exposing a `PokiSDK` global: `init(): Promise` Ôćĺ `gameLoadingFinished()` Ôćĺ `gameplayStart()`/`gameplayStop()` Ôćĺ `commercialBreak(onStart?): Promise` / `rewardedBreak(opts?): Promise<boolean>`. Hard requirements: **audio and keyboard input disabled during breaks**; **spacebar/arrow default scroll prevented**. **Implementation: a typed `PortalSdk` interface with a `NullPortalSdk` (dev/test ÔÇö the game is fully functional without the SDK, Poki's own recommended local mode) and a `PokiPortalSdk` adapter binding the CDN global; the ad-break flow contractually wraps audio-mute + input-suspend and is test-proven against a fake.** The official npm typed wrapper `@poki/sdk@0.0.4` (Poki-maintained, June 2026) was weighed and **rejected**: a pre-1.0 forwarder around the very global we wrap behind an injected interface anyway (dm-0086).
10. **Package policy (dm-0038 continuation) ÔÇö re-verified at plan time (2026-07-11) against registry.npmjs.org:** `typescript` 7.0.2 and `@types/node` 26.1.1 are the exact latest stable releases; both already pinned. **P9 adds zero new npm dependencies, runtime or dev** ÔÇö raw WebGL2/WebAudio/Canvas2D (all in `lib: ["DOM"]`, already in tsconfig), hand-rolled fakes, CDN-loaded Poki SDK. Deliberate, not an omission: every candidate (jsdom, headless-gl, pixi/three, @poki/sdk) weighed and rejected in dm-0086 with reasons.
11. **Advanced techniques ÔÇö adopt what the REQs and the bible name, architect seams for what P11 needs, build nothing speculative (dm-0082/dm-0086).** Adopted: **cache-by-key rasterization** (`kind:state:size`, ~6px jitter pad, level-scoped, cleared on transition ÔÇö bible ┬ž5); **states are frames** (separately cached bitmaps swapped by key, never runtime tinting ÔÇö bible ┬ž4); **one composed background texture per level** with UV-crop camera (tile polygons painted into one bitmap ÔÇö dm-0080's seamless-mass rule ÔÇö then uploaded once); **texture-atlas packing** of dynamic-entity rasters (shelf packer, page size from profile data); **single-draw-call sprite batching** keyed by `(atlasPage, blendMode)`; **WebGL2 instanced quads** for dynamics/particles; **static/dynamic split at load** (bible ┬ž5); **viewport culling** through the P3 `SpatialPartition` quadtree consumed read-only (REQ-162's own wording ÔÇö reuse, not a second tree); **fixed-step interpolation** via `interpolationAlpha`; **generation-counted object pools** (REQ-161) sized from profile data; **DPR cap** (2 gameplay / 1.5 sheets ÔÇö bible ┬ž5); **hysteresis + cooldown quality-tier controller** over a frame-time EMA (REQ-163), tiers as data. Rejected: dirty-rect partial redraw (bookkeeping beats a batched full-frame redraw at this scale), per-frame procedural drawing in any form. **Fairness ├Ś quality interlock:** the scale-back controller is *forbidden from degrading grammar-critical signatures* ÔÇö REQ-016 says failure information is always visually present, so every `DrawList` item carries a `critical` flag derived from its grammar category (Danger is always critical) and the tier controller may only drop non-critical items (decor, grain layer, particles). Cross-REQ interlock as a validation criterion, not a convention.

## Design summary (the compact model the code is written from)

1. **Grammar (`render/grammar/Grammar.ts`)** ÔÇö style-agnostic, versioned, strict-parsed data: the six `GrammarCategory` ids (`safe`, `danger`, `interactive`, `temporary`, `optimization`, `secret`), `categoryBindings` (`behavior.kind`/structural role Ôćĺ exactly one category; **goal Ôćĺ interactive** per dm-0085), per-category `critical: boolean` (danger structurally always-critical), and the signature-role contract each pack must fill (palette ┬Ě silhouette ┬Ě motion ┬Ě audio). Parser rejects unknown categories, unbound kinds, double bindings, non-critical danger. `DEFAULT_GRAMMAR` ships complete.
2. **StylePack seam (`render/style/StylePack.ts`)** ÔÇö the dm-0076 asset-provider interface: `visual(request: {category, kind, state, sizePx, identitySeed}) Ôćĺ CachedVisual` (bitmap handle + anchor + pad), `composeBackground(level, decorSeed) Ôćĺ BackgroundBitmap`, `audioPatch(category) Ôćĺ SynthPatch`, `motionParams(category) Ôćĺ MotionSpec`, plus `validatePack(pack, grammar)` enforcing one complete, non-overlapping signature per category (terracotta outside Danger ÔÇö e.g. optimization glyphs ÔÇö is a validation error per dm-0084). Cache-by-key with generate-never-per-frame semantics lives *behind* this seam.
3. **PaperStylePack (`render/style/paper/`)** ÔÇö StylePack #1 (dm-0075): a TypeScript port of the reference engine's committed core ÔÇö seeded identity-hash RNG, rough-polygon edge walker, the 9-slice `blockSlab` library (dm-0078: `base(kind)` byte-identical across states + `overlay(kind,state)`), the `PaperTiles` modular terrain kit painted into one background bitmap (dm-0080: clean solid interiors, external-boundary-only tearing, 4-neighbour autotile, out-of-bounds-counts-as-solid), the 8-type decor set + single shared grain layer (dm-0077), and the six category signatures of bible ┬ž1 as amended by dm-0084/dm-0085. Draws only through the injected `Raster2D` device.
4. **Scene compiler (`render/scene/SceneCompiler.ts`, `render/scene/Camera.ts`)** ÔÇö pure: `compileScene(state, previous, alpha, grammar, packHandles, viewport) Ôćĺ DrawList`. Interpolates transforms between snapshots by `alpha`; resolves every visible entity through the grammar to its pack visual (one category Ôćĺ all four signatures); culls to the camera's world-space AABB via the P3 quadtree (read-only); emits plain-data draw items (background UV crop, static runs, instanced dynamics, particles, overlays) each tagged `critical` per category. The camera (position smoothing toward the player, REQ-150 share) is render-layer state, never written back.
5. **GL executor + batcher (`render/gl/Gl2Device.ts`, `render/gl/Batcher.ts`, `render/gl/GlRenderer.ts`, `render/gl/Atlas.ts`)** ÔÇö `Gl2Device` is the minimal typed subset of `WebGL2RenderingContext` used (enumerated, documented); `Atlas` shelf-packs cached rasters into pages (size from profile data) and uploads once per level; `Batcher` folds a `DrawList` into (a) the background quad, (b) per-frame instanced dynamic buffers grouped by `(atlasPage, blend)`, minimizing draw calls; `GlRenderer` replays batches against the device. Tests drive a recording `FakeGl2` and assert call sequences + buffer contents + upload-once behavior.
6. **Pools (`render/pool/Pool.ts`, `render/pool/ParticlePool.ts`)** ÔÇö generic fixed-capacity, generation-counted pool (REQ-161: particles, visual impacts, projectiles); stale handles detected by generation mismatch; capacities from profile data, not literals.
7. **Game feel (`render/feel/`)** ÔÇö anticipation frames, squash-and-stretch, jump particle burst, landing impact (REQ-150 visual sub-clauses) as pure functions of observable state transitions (jump-lock phase changes, grounded transitions), emitted as DrawList modifiers/pool spawns; the defeat marker (REQ-016) renders the death cause at the defeat position ÔÇö always `critical`. Animation curves are data (`FeelProfile`), not literals.
8. **Audio (`render/audio/`)** ÔÇö `AudioDevice` seam; `deriveAudioCues(previous, current) Ôćĺ AudioCue[]` (pure, transition-driven: jump, defeat, goal, checkpoint, surface class); `PatchCompiler` turns a category's procedural `SynthPatch` (waveform, freq/gain envelopes, duration ÔÇö pack material, zero binary assets in P9) into a node-graph plan the executor schedules. Master gain exposes `muteForBreak()`/`resume()` for the ad flow.
9. **Assets (`render/assets/`)** ÔÇö manifest schema (strict-parsed): `id`, `url`, `kind` (`level`, `grammar`, `pack-calibration`, later `audio-sample`/`atlas-image`), `priority` (`critical`|`deferred`), `bytes`. `AssetLoader` over `AssetFetcher`: critical tier gates first render; deferred tier streams after; progress feeds the shell's loading surface and `gameLoadingFinished`; per-asset delivery timing (bytes/elapsed via injected clock) plus rasterize/upload timing feeds a `DeliveryReport` ÔÇö REQ-132's P9 share, same diagnostic-only contract as dm-0067.
10. **Shell (`render/shell/`, `render/platform/`)** ÔÇö `GameShell` owns the loop: `FrameScheduler` tick Ôćĺ `Clock.advance(clockState, realDelta)` Ôćĺ N ├Ś fixed steps through the existing `Engine` Ôćĺ `compileScene(...)` Ôćĺ `GlRenderer.draw` + audio cues; `ViewportModel` (pure) maps window size/DPR/fullscreen to canvas size + world-to-screen transform with letterboxing and the DPR caps (REQ-171); `InputCapture` maps key/pointer events to the existing `InputFrame` shape and flags default-prevention for space/arrows; `PortalSdk` lifecycle: `init` Ôćĺ loader progress Ôćĺ `gameLoadingFinished` Ôćĺ `gameplayStart` on first input / `gameplayStop` on defeat/goal Ôćĺ `commercialBreak` wrapping mute + input-suspend. `render/platform/` holds the only browser-global bindings.
11. **Editor/debug UI (`render/tooling/`)** ÔÇö `paintOverlays(descriptorsÔÇŽ) Ôćĺ DrawList` for all six P8 descriptor kinds; `EditorSurface`: `LevelDraft Ôćĺ DrawList` (grid, entities, selection) + `mapEditorInput(pointer/key events) Ôćĺ EditorCommand` feeding P8's `applyCommand`/`undo`/`redo` unchanged; inspector bindings expose pause/step/reload/variable-edit as UI commands calling P8's `Inspector` verbatim. `render/tooling/` is the *only* `render/` area importing `tools/` (dm-0081).
12. **Quality controller (`render/quality/`)** ÔÇö pure: frame-time EMA over shell-reported durations; tier ladder from profile data (full Ôćĺ reduced-particles Ôćĺ no-deferred-decoration Ôćĺ no-grain), transitions guarded by hysteresis + cooldown; `applyTier(drawList, tier)` may drop only `critical: false` items (the REQ-016 interlock); scale-up re-tests after a stability window. Verified under synthetic load (scripted frame-time sequences) ÔÇö the IRD exit condition.

## Adversarial review ÔÇö findings and upgrades over the naive slice table

1. **Where does authored content *live*? (never asked by the naive table.)** `docs/directory_structure.md` names no directory for persisted level/chapter/campaign data ÔÇö `src/gen/` is generation *logic*, `src/eval/campaign/` holds campaign *runtime record types*, `src/schema/` holds the level/tape/profile schemas but no chapter/campaign authoring schema exists anywhere. Every existing home fails: putting JSON payloads inside `src/gen/` conflates design-time logic with design-time output and gives the isolation scan a new, unintended class of file to reason about; putting them in `src/eval/campaign/` couples authored data to the runtime-record types that *consume* it, which is backwards (data outlives any one runtime shape); `src/schema/` is reserved for versioned wire-format types with no content of their own. **Fix (dm-0108): a new top-level `content/` subtree**, one-way-isolated exactly like `src/gen/`/`tools/`/`render/` (dm-0057/dm-0066/dm-0081) ÔÇö `content/` may import `src/eval/` only through its named public seams (`evaluateLevel`, `Evaluate.assembleLevelEvidence`, `AgentHarness.runAgent`/`CANONICAL_PIPELINE`, `campaign/`'s `processCampaign`/`updateState`, `macro/Curriculum`'s `validateCurriculum`) and `src/gen/` only through its named public seams (`manufactureLevel`, `Concept.validateConcept`, `Creativity.evolveLevel`); **nothing under `src/` may ever import `content/`**; `render/` and `tools/` do not import `content/` either in P10 (the P8 editor's touch-up path, if ever wired to `content/`, is out of scope here ÔÇö noted as an open question). `content/data/` holds the persisted JSON payloads (chapter framework files, campaign manifest, generated `LevelDefinition` files); `content/*.ts` holds the schema + assembler + estimator logic that produces and validates them. A `ContentIsolation.test.ts` scan lands at S10.1, following the exact `<Subtree>Isolation.test.ts` convention of the four prior subtrees. `directory_structure.md` and `tsconfig.json`'s `include` gain the subtree at S10.1.
2. **The REQ-083 seven-step framework has no schema, so "author it" has no acceptance criterion.** REQ-083 demands Theme/Learning Goal/Mental Model/Misconceptions/Subversion/Optimization/Final Exam be defined *before* any level generation, but nothing enforces "before" or even "all seven present" today. **Fix (dm-0109): `content/schema/ChapterFramework.ts` ÔÇö the eighth versioned, strict-parsed schema** (after `LevelDefinition`, `ScoringProfile`, `CampaignProfile`, `GenProfile`, `TelemetryProfile`, `AssetManifest`, and `Grammar`/`StylePack` data ÔÇö the project's ninth is closer to accurate once P9's grammar schema is counted, but the count itself is not load-bearing; what matters is it follows the same discipline). Fields: `frameworkSchemaVersion`, `chapterId`, `theme: string`, `learningGoal: string`, `mentalModel: string`, `misconceptions: readonly string[]` (Ôëą1), `subversion: string`, `optimizationFocus: string`, `finalExam: string`, and `levelSlots: readonly ChapterLevelSlot[]` where each slot fixes `slotId`, `archetype` (one of `LEVEL_ARCHETYPES`, dm-0110 below), `targetDifficultyTier` (REQ-084 bucket, dm-0111), `arcPhase` (one of `EMOTION_ARC`'s six phases, structurally reusing `DesignSpace.ts`'s existing type rather than inventing a parallel one), and `seed: number`. `parseChapterFramework` rejects: any of the seven textual fields empty, fewer than one misconception, an `arcPhase`/`archetype` value outside the closed sets, a `levelSlots` array with duplicate `slotId`s. **The pipeline's ordering is enforced structurally, not by convention: `manufactureLevel` for a slot is only ever called with a `LevelConcept` built by `content/Assembler.ts` from an already-parsed `ChapterFramework`'s slot ÔÇö there is no code path to generate a level without a validated framework upstream of it**, which is what "before any level generation" actually requires as a proof rather than a promise.
3. **REQ-100 (Dual-Path) has zero code presence and, read carelessly, looks like it needs a whole new mechanism.** Nothing in `src/` names discovery/mastery paths. But `src/eval/local/Optimization.ts`'s `OptimizationVerdict` already computes exactly the evidence dual-path-ness requires: `applicable: boolean` (both a discovery completion and a search/archetype-derived optimum exist) and `deltaSeconds` (`Time_Discovery Ôłĺ Time_WorldRecord`, REQ-102's own metric). **Fix (dm-0110): REQ-100 is proven by composition, not new code.** A level satisfies the Dual-Path axiom iff its `OptimizationVerdict.applicable === true` **and** `deltaSeconds >= profile.dualPath.minDeltaSeconds` (a new calibrated field, not a literal) ÔÇö the same physical level supports a slow-but-legible discovery route and a materially faster optimized route, which is precisely what "two paths in one space" means operationally for a one-jump precision platformer (there is no second exit or branching geometry to invent; the *paths* are execution-quality routes through the same geometry, matching REQ-005's "depth via level design, not expanded power sets"). `content/DualPathProof.ts` exports one pure function, `proveDualPath(verdict, profile): DualPathVerdict`, wrapping this check with a typed reason string on failure. Rejected: authoring literal alternate physical routes per level (violates REQ-005's explicit rejection of power-set depth expansion, and duplicates geometry the one-jump constraint is designed to keep singular); inventing a new simulation-side path-counting mechanism (re-derives what `Optimization.ts` already proves ÔÇö a repeat of the dm-0107 lesson, never re-derive classification logic that a lower layer already computed).
4. **REQ-084's difficulty distribution has no source of truth, and two unrelated "tier" concepts risk being conflated.** `TierTimes` (`discovery`/`good`/`fast`/`expert`/`worldRecord`) are **routing** tiers ÔÇö measured time-based skill anchors within *one* level (REQ-101). `DifficultyAxis`/`difficultyVectors` are an **authored [0,1] intent vector**, explicitly marked provisional (dm-0015) and never yet validated against measured outcomes. `CurriculumLevel.difficulty` is an **unspecified scalar the caller must supply** ÔÇö a placeholder the naive plan could easily mistake for "already solved." None of the three is the REQ-084 Easy/Medium/Hard/Harder/Very-Hard **campaign-bucket** a level belongs to. **Fix (dm-0111): `content/DifficultyEstimator.ts` derives the REQ-084 tier from measured evidence, never from the authored intent vector alone** ÔÇö `estimateDifficultyTier(evidence: EvidenceBundle, verdict: OptimizationVerdict, profile: DifficultyProfile): DifficultyTier`, a weighted, calibrated function of (a) `TierTimes.discovery` (raw completion time, REQ-101), (b) agent attempt/failure counts from the `EvidenceBundle`'s kill-switch data (execution demand), (c) `deltaSeconds` (optimization headroom ÔÇö REQ-102), with the *authored* `difficultyTarget` used only as the generator's aim point (`Generator.ts` already consumes it, unchanged) and never substituted for the measured tier. Weights and bucket boundaries live in a new `DifficultyProfile` (calibrated data, per REQ-084's own "calibrated data, not hardcoded" wording), not literals. `CurriculumLevel.difficulty`'s open placeholder is resolved by feeding it this same estimated tier's numeric value (documented at the S10.2 slice, a read of `Curriculum.ts`'s exact call site required before wiring ÔÇö zero-assumption, not assumed here). REQ-084's percentage bands (20/35/25/15/5) are enforced at campaign-assembly time (S10.5) by counting `DifficultyTier` occurrences across all authored slots, not per-level.
5. **The naive table's S10.2 is "dual-path level generation" as a single slice ÔÇö sized as if generating the entire campaign's levels fits one session.** Every prior content-adjacent phase (P7's manufacturing pipeline, P8's tooling) took 6ÔÇô8 slices to build the *mechanism*; generating dozens of real levels through that mechanism, then discovering the GDOS thresholds were calibrated for the synthetic fixtures they were tested against (dm-0031's profile was never run against real content), is qualitatively riskier than building the mechanism was. **Fix: pilot-then-batch.** S10.3 manufactures exactly **one pilot chapter** end-to-end (framework Ôćĺ generation Ôćĺ evaluation Ôćĺ assembly Ôćĺ macro validation) and treats any GDOS/macro-criteria threshold miss as a **calibration signal**, not a content bug ÔÇö the CDRE profile-evolution loop built at P5 exists for exactly this, and a threshold change here is ledgered exactly like the dm-0084/dm-0085 grammar reconciliations were, never silently overridden per-level (the DoD's explicit prohibition). S10.4 batches the remaining chapters only after the pilot's thresholds hold, splittable into sub-sessions by chapter without a plan amendment (chapter count is data, decided at S10.1 from the live mechanic-lifecycle registry ÔÇö see finding 8 ÔÇö not guessed here).
6. **REQ-090's "Campaign Integration" pipeline phase is prose, not code.** `PipelinePhase` has seven members, not eight ÔÇö "Targeted Revision" folds into the S10-irrelevant evolutionary loop between phases 3/4 (already correct, P7's business), but "Campaign Integration" (phase 8 in the REQ-090 prose) corresponds to nothing: `manufactureLevel`'s accept branch returns a `PipelineProduct` and stops. Assuming this phase already writes into a campaign structure would have been exactly the kind of unread-file guess the zero-assumption invariant forbids. **Fix (dm-0112): `content/Assembler.ts::integrateProduct(product: PipelineProduct, chapterId: string): LevelRecord`** is the literal, previously-missing REQ-090 phase 8 ÔÇö it is P10 code, not a P7 retrofit, because P7 closed VERIFIED on the seven phases it actually owns (dm-0064) and this integration step only becomes meaningful once real chapters exist to integrate into. 10. **Generation mode ÔÇö the actual creative-authoring question, not an engineering one.** Three modes were weighed: (a) fully procedural (`manufactureLevel` runs unattended per slot, zero human/agent creative input beyond the framework); (b) hand-authored via the P8/P9 editor, then validated; (c) **overgenerate-and-curate** ÔÇö `Creativity.evolveLevel`'s existing REQ-081 loop runs several candidates per slot, all pass through `evaluateLevel`, the highest-scoring accepted candidate is selected, and the selection itself is ledgered. **Chosen: (c) (dm-0114).** REQ-050 ("all creative decisions originate from GDOS; no gameplay content created independently of it") is close to unsatisfiable under mode (b) as the primary path ÔÇö an editor-authored level's *creative* content would originate from a human/session's taste, with GDOS only checking it after the fact ÔÇö and mode (c) is exactly what the P7 creativity loop and P5 CDRE were built to feed. The P8/P9 editor remains available as a **touch-up path** for a candidate that fails a gate by a narrow margin, but any touch-up re-enters the full `evaluateLevel` pipeline before acceptance ÔÇö it never bypasses scoring. **Determinism corollary:** every accepted level's generation seed is part of `PipelineProduct.provenance.seed` already (S9-independent, already built); `content/Assembler.ts` persists that seed into the level's `content/data/` payload so campaign regeneration from the same `ChapterFramework` + seed set is byte-identical and idempotent (the protocol's idempotency invariant, applied to content for the first time).
11. **Two structural gaps worth closing now while they're cheap.** (a) `arcPhase` lands on every `ChapterLevelSlot` (finding 2) specifically so REQ-015's six-phase arc audit at macro validation time is a **structural** check over authored data (every phase present, in order, across a chapter) rather than a convention nobody enforces ÔÇö this closes the gap the P9 handoff explicitly deferred to P10. (b) Campaign size (finding 8) is fixed at S10.1 from live registry data specifically so REQ-084's percentage math has a stable denominator before any level is generated, rather than being recomputed (and potentially redefining "5%" of a moving total) mid-campaign.
